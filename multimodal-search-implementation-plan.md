# Multimodal Gameplay Search — Implementation Plan

This is the end-to-end build plan for the search pipeline described in the design doc. It assumes the existing work you already have:

- **MediaPipe body detection** is running and producing an `armStates` stream.
- **Punch tracking** is in progress inside that stream.
- **Recorder** produces 5-second WebM chunks and emits `ChunkReady` events.

Stack decisions based on your answers:

- Backend: **Next.js API routes + Supabase** (Postgres, Storage, Webhooks). No Supabase Auth — players use anonymous text IDs generated client-side.
- Embeddings: **Gemini `text-embedding-004`** at **768 dimensions**.
- Captioning: **Gemini** (multimodal, for video understanding).
- Async embedder: **Next.js API route invoked by a Supabase Database Webhook**, with the actual work offloaded to a background task using Next.js `after()`. Rationale below in Phase 4.

Key ID conventions (must stay consistent with existing `rooms` / `room_players` schema):
- `player_id` — **`text`** everywhere. Player IDs are anonymous strings generated by `useIdentity()`. Not UUIDs, not Supabase Auth UIDs.
- `match_id` — **`uuid`** generated server-side when a match starts. A room can host multiple sequential matches, so `match_id ≠ room_id`. The room's UUID is stored alongside the match row for joining but is not the primary key.
- `room_id` — **`uuid`** referencing `rooms.id`, carried on the match row for context.

The plan is structured in phases. Each phase is independently testable and leaves the system in a working state. Do not skip ahead — later phases assume earlier invariants.

## Phase 0 — Foundations (schema, storage, env)

Before writing any feature code, get the database, storage, and config in place. Everything downstream depends on this being right.

### 0.1 Enable extensions

In the Supabase SQL editor:

```sql
create extension if not exists vector;
create extension if not exists pg_trgm;
```

`vector` is pgvector for embeddings. `pg_trgm` is optional but useful later if you want fuzzy text search on captions as a fallback.

### 0.2 Create the three core tables

The schema mirrors the design doc exactly — three tables with clearly separated roles.

```sql
-- Matches: one row per game session within a room. A room can host multiple matches.
create table matches (
  id          uuid primary key default gen_random_uuid(),
  room_id     uuid not null references rooms(id) on delete cascade,
  started_at  timestamptz not null default now(),
  ended_at    timestamptz,
  status      text not null default 'active' check (status in ('active', 'finished'))
);

create index matches_room_idx on matches (room_id);

-- Every discrete action event as it happened. Authoritative timeline.
-- Named ActionEvent in client code to avoid collision with the existing
-- GameEvent type in lib/multiplayer/types.ts (which is for room signaling).
create table match_events (
  id            bigserial primary key,
  match_id      uuid not null references matches(id) on delete cascade,
  player_id     text not null,              -- text, not uuid — matches room_players.player_id
  event_type    text not null,              -- 'punch', 'dodge', 'kick', etc.
  event_subtype text,                       -- 'jab', 'cross', 'hook', etc.
  occurred_at   timestamptz not null,       -- wall-clock time the event fired
  match_time_ms integer not null,           -- ms since match start
  metadata      jsonb not null default '{}'::jsonb
);

create index match_events_match_idx    on match_events (match_id, match_time_ms);
create index match_events_player_idx   on match_events (player_id, occurred_at desc);
create index match_events_type_idx     on match_events (event_type);

-- The unit of video search. One row per 5-second chunk.
-- ended_at is derived on the client as new Date(startedAt + durationMs).toISOString()
-- since ChunkReady only carries startedAt + durationMs (no separate endedAt field).
create table clips (
  id               uuid primary key default gen_random_uuid(),
  match_id         uuid not null references matches(id) on delete cascade,
  player_id        text not null,           -- text, not uuid
  chunk_index      integer not null,
  storage_path     text not null,           -- path in Supabase Storage bucket 'gameplay-clips'
  started_at       timestamptz not null,
  ended_at         timestamptz not null,
  duration_ms      integer not null,
  event_counts     jsonb not null default '{}'::jsonb,  -- {"punch": 4, "dodge": 1}
  caption          text,                    -- populated async by embedder
  embedding        vector(768),             -- Gemini text-embedding-004
  embedding_status text not null default 'pending' check (embedding_status in ('pending','processing','ready','failed')),
  created_at       timestamptz not null default now(),
  unique (match_id, player_id, chunk_index)
);

create index clips_match_idx          on clips (match_id);
create index clips_player_idx         on clips (player_id, started_at desc);
create index clips_event_counts_idx   on clips using gin (event_counts jsonb_path_ops);
create index clips_embedding_idx      on clips using ivfflat (embedding vector_cosine_ops) with (lists = 100);
create index clips_status_idx         on clips (embedding_status) where embedding_status != 'ready';

-- Per-match rollups written at match end.
create table match_summaries (
  match_id        uuid primary key references matches(id) on delete cascade,
  player_id       text not null,            -- text, not uuid
  started_at      timestamptz not null,
  ended_at        timestamptz not null,
  duration_ms     integer not null,
  event_totals    jsonb not null default '{}'::jsonb,  -- {"punch": 87, "dodge": 12}
  best_combo_len  integer,
  created_at      timestamptz not null default now()
);

create index match_summaries_player_idx on match_summaries (player_id, started_at desc);
```

A few notes on the choices:

The GIN index on `event_counts` with `jsonb_path_ops` makes queries like `event_counts @> '{"punch": 5}'` fast, but that operator only checks containment, not numeric comparisons. For `>` and `>=` comparisons you'll use expressions like `(event_counts->>'punch')::int >= 5`, which won't hit the GIN index but is fine at the volumes we're dealing with — a player's clips table will be in the thousands, not millions. If it becomes hot, add a generated column for `punch_count` with a btree index.

The ivfflat vector index with 100 lists is a reasonable default for tens of thousands of clips. Don't bother tuning it until you have real data. Remember to `ANALYZE clips` after bulk inserts so the planner knows what it's dealing with.

`embedding_status` lets the embedder worker claim rows safely (pending → processing → ready/failed) so you can retry failures without double-processing.

### 0.3 Storage bucket

Create a private Supabase Storage bucket called `gameplay-clips`. Keep it private — clips will be served via signed URLs generated at query time. Set a reasonable file-size policy (something like 20 MB per chunk is more than enough headroom for 5-second WebM).

Path convention:

```
gameplay-clips/{match_id}/{player_id}/{chunk_index}.webm
```

This is predictable, match-scoped, and makes deletion trivial if a user asks you to wipe a match.

### 0.4 Environment variables

In `.env.local` and your Vercel project:

```
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
SUPABASE_SERVICE_ROLE_KEY=...       # server-only, used by API routes
GEMINI_API_KEY=...
EMBEDDER_WEBHOOK_SECRET=...         # shared secret for the webhook endpoint
```

Never expose the service role key to the client. All writes happen through API routes running server-side.

### 0.5 Row Level Security

Turn RLS on for all tables. All writes go through API routes using the service role key, which bypasses RLS. There is no Supabase Auth in this project, so `auth.uid()` is never available — RLS policies cannot filter by player identity on the client side. Keep it simple for now: enable RLS to block direct anonymous access, and rely on the service role for all reads/writes from API routes.

```sql
alter table matches         enable row level security;
alter table match_events    enable row level security;
alter table clips           enable row level security;
alter table match_summaries enable row level security;

-- No client-facing select policies yet — all access is via service-role API routes.
-- When adding a results/replay page, add policies scoped to the room's player list
-- rather than auth.uid(), since players are identified by text ID, not Supabase Auth.
```

## Phase 1 — EventTracker (client-side event detection)

This is the piece that produces the structured half of the multimodal data. Get this solid before touching ingestion, because everything downstream trusts its output.

### 1.1 Shape of the output

Two outputs, both driven by the same detection logic:

```ts
// Named ActionEvent (not GameEvent) to avoid collision with the existing
// GameEvent type in lib/multiplayer/types.ts, which is used for room
// signaling (game_start / game_end / player_ready).
type ActionEvent = {
  type: 'punch' | 'dodge' | 'kick' | string;
  subtype?: string;               // 'jab' | 'cross' | 'hook' | ...
  occurredAt: number;             // Date.now() when detected
  matchTimeMs: number;            // ms since match start
  metadata: Record<string, unknown>;
};

type ChunkRollup = {
  chunkIndex: number;
  counts: Record<string, number>;  // { punch: 4, dodge: 1 }
};
```

The **event log** is the ground truth written to `match_events`. The **chunk rollup** is derived from the same events, scoped to a chunk window, and is what gets attached to a clip at upload time.

### 1.2 Detection heuristics

You're already doing punch detection. Extend the same pattern for other gestures. The general shape of a detector:

1. Subscribe to the latest `armStates` frame.
2. Maintain a small state machine per limb (idle → extending → extended → retracting → idle).
3. When the state machine completes a valid cycle within plausible time bounds and with plausible velocity, emit an event.

Keep detectors independent and composable — one file per gesture type. This pays off when you add kicks, dodges, blocks, etc. Don't try to unify them behind a single "gesture engine" abstraction yet; that's premature.

### 1.3 EventTracker class

```ts
class EventTracker {
  private events: ActionEvent[] = [];
  private currentChunkCounts: Record<string, number> = {};
  private currentChunkIndex = 0;
  private matchStartedAt: number;

  constructor(opts: { matchStartedAt: number }) {
    this.matchStartedAt = opts.matchStartedAt;
  }

  ingestFrame(frame: ArmStatesFrame) {
    const detected = runAllDetectors(frame);   // returns ActionEvent[]
    for (const ev of detected) this.record(ev);
  }

  private record(ev: ActionEvent) {
    this.events.push(ev);
    this.currentChunkCounts[ev.type] = (this.currentChunkCounts[ev.type] ?? 0) + 1;
  }

  /** Called by the Recorder whenever a chunk boundary is crossed. */
  rollChunk(nextChunkIndex: number): ChunkRollup {
    const rollup = {
      chunkIndex: this.currentChunkIndex,
      counts: this.currentChunkCounts,
    };
    this.currentChunkIndex = nextChunkIndex;
    this.currentChunkCounts = {};
    return rollup;
  }

  /** Drains queued events for batched upload. */
  drainEvents(): ActionEvent[] {
    const out = this.events;
    this.events = [];
    return out;
  }
}
```

Two independent outputs from one stream. The rollup is per-chunk and gets attached at upload time. The drained event list is flushed on a separate cadence (every 2–3 seconds is fine) to `/api/match-events`.

### 1.4 Wiring to the Recorder

The Recorder already emits `ChunkReady`. Hook the boundary:

```ts
recorder.subscribe((chunk) => {
  const rollup = eventTracker.rollChunk(chunk.chunkIndex + 1);
  uploadChunk({ chunk, rollup });   // Phase 2
});
```

A subtle timing issue: `ChunkReady` fires when a chunk finishes encoding, which is slightly after the wall-clock boundary. That's fine — as long as the rollup is snapshotted at the same moment you receive `ChunkReady`, the counts belong to that chunk by construction. Don't try to align to exact timestamps.

### 1.5 Testability

Write unit tests that feed a synthetic `armStates` sequence (scripted joint positions) into the detectors and assert the right events come out. This is the kind of code that silently rots if you change the detection constants without tests, and silent rot in the authoritative ground-truth layer will poison everything above it.

## Phase 2 — Ingestion (chunk upload + event log)

Two endpoints. Both are Next.js API routes using the Supabase service role client.

### 2.1 `/api/clips/upload` — chunk upload

Accepts the video blob plus its rollup. Does three things in order:

1. Upload blob to Supabase Storage at `{match_id}/{player_id}/{chunk_index}.webm`.
2. Insert a row into `clips` with `storage_path`, `event_counts`, timing, and `embedding_status = 'pending'`.
3. Return the clip row id.

Use `multipart/form-data` for the upload — a field for the blob, a JSON field for the metadata.

Important notes:
- Use the **service role key** here, not the anon key. The existing `lib/supabase/server.ts` uses the anon key and is intended for auth-scoped reads. API routes that write need `createClient(url, SUPABASE_SERVICE_ROLE_KEY)` directly.
- `ChunkReady` has `startedAt` and `durationMs` but no `endedAt`. Derive `ended_at` on the client: `new Date(chunk.startedAt + chunk.durationMs).toISOString()`.
- Storage bucket is `gameplay-clips` (private, created in Phase 0.3).

```ts
// app/api/clips/upload/route.ts
import { createClient } from '@supabase/supabase-js';

export const runtime = 'nodejs';         // we need Buffer for the blob
export const maxDuration = 30;

export async function POST(req: Request) {
  const form = await req.formData();
  const blob = form.get('chunk') as Blob;
  const meta = JSON.parse(form.get('meta') as string);

  // Service role key — bypasses RLS, never sent to the client.
  const supabase = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!
  );

  const path = `${meta.matchId}/${meta.playerId}/${meta.chunkIndex}.webm`;

  const { error: upErr } = await supabase.storage
    .from('gameplay-clips')
    .upload(path, blob, { contentType: 'video/webm', upsert: true });
  if (upErr) return Response.json({ error: upErr.message }, { status: 500 });

  const { data, error: insErr } = await supabase
    .from('clips')
    .insert({
      match_id:      meta.matchId,
      player_id:     meta.playerId,
      chunk_index:   meta.chunkIndex,
      storage_path:  path,
      started_at:    meta.startedAt,
      ended_at:      meta.endedAt,    // client computes: new Date(startedAt + durationMs).toISOString()
      duration_ms:   meta.durationMs,
      event_counts:  meta.rollup.counts,
    })
    .select('id')
    .single();
  if (insErr) return Response.json({ error: insErr.message }, { status: 500 });

  return Response.json({ clipId: data.id });
}
```

The row is immediately useful for count-based queries. The caption and embedding fields stay null until the embedder processes it (Phase 4).

### 2.2 `/api/match-events` — batched event log

Accepts an array of `ActionEvent` objects, flushed from the client every few seconds. Bulk-inserts to `match_events`.

```ts
export async function POST(req: Request) {
  const { matchId, playerId, events } = await req.json();

  const rows = events.map((e: ActionEvent) => ({
    match_id:      matchId,
    player_id:     playerId,
    event_type:    e.type,
    event_subtype: e.subtype ?? null,
    occurred_at:   new Date(e.occurredAt).toISOString(),
    match_time_ms: e.matchTimeMs,
    metadata:      e.metadata ?? {},
  }));

  const { error } = await supabase.from('match_events').insert(rows);
  if (error) return Response.json({ error: error.message }, { status: 500 });
  return Response.json({ ok: true });
}
```

Idempotency: events can duplicate if a flush is retried. If you care, add a client-generated `event_id` UUID to `GameEvent` and make it a unique column. For a first cut, you can skip this — occasional duplicates won't materially distort counts.

### 2.3 Match lifecycle

Add `/api/matches/start` and `/api/matches/end`.

**Start** inserts a row into the new `matches` table (which references `rooms.id`) and returns the generated `match_id`. The client stores this UUID for the session — it gets passed to the recorder's `start(matchId, playerId)` call and included in every chunk upload and event flush.

**End** closes the match row and writes the `match_summaries` row by aggregating `match_events`:

```sql
insert into match_summaries (match_id, player_id, started_at, ended_at, duration_ms, event_totals)
select
  match_id,
  player_id,
  min(occurred_at),
  max(occurred_at),
  extract(epoch from (max(occurred_at) - min(occurred_at))) * 1000,
  jsonb_object_agg(event_type, cnt)
from (
  select match_id, player_id, event_type, count(*) as cnt
  from match_events
  where match_id = $1
  group by match_id, player_id, event_type
) t
group by match_id, player_id;
```

Do this server-side in the `end` route. Don't try to maintain `match_summaries` incrementally — write-on-end is simpler and correct.

## Phase 3 — Recorder ↔ ingestion wiring (client)

You already have the Recorder. This phase just wires it to Phase 2.

### 3.1 Upload on ChunkReady

```ts
recorder.subscribe(async (chunk) => {
  const rollup = eventTracker.rollChunk(chunk.chunkIndex + 1);

  }));

  await fetch('/api/clips/upload', { method: 'POST', body: form });

```

### 3.2 Event log flush

Separate timer, every 2–3 seconds:

```ts
setInterval(async () => {
  const events = eventTracker.drainEvents();
  if (!events.length) return;
  await fetch('/api/match-events', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ matchId, playerId, events }),
  });
}, 2500);
```

### 3.3 Failure handling

Network blips will happen. Keep a small in-memory retry queue on the client for both chunks and event batches. On upload failure, stash the payload and retry on the next tick with exponential backoff. Don't block new uploads on old failures — each chunk is independent.

At this point you have a working system for count-based queries. You can already answer "how many punches this session" with zero AI involved. Validate this phase end-to-end before moving on.

## Phase 4 — Embedder (async captioning + embedding)

This is the second modality. It runs out-of-band and adds semantic searchability to clips over time.

Note: `lib/embeddings/index.ts` and `lib/generation/index.ts` already exist as empty stubs. Fill those out in this phase rather than creating new files — `lib/generation/` for the Gemini client wrapper, `lib/embeddings/` for the embed helper.

### 4.1 Why Next.js API route + `after()`

You asked for a recommendation. Here's the tradeoff:

- **Supabase Edge Functions** are fine but add a second runtime to reason about, and calling Gemini from Deno is more awkward than from Node.
- **Separate Node worker + queue** is the "right" long-term answer but is overkill for now — it adds deploy surface area you don't need yet.
- **Next.js API route as webhook target, using `after()` for the actual work** keeps everything in one stack, scales on Vercel, and is trivially replaceable by a real queue later without changing the webhook contract.

Use Next.js's built-in `after()` from `'next/server'` (available in Next.js 15+) — not the legacy `globalThis.waitUntil` pattern. The `after()` function runs after the response is sent, keeping the function alive for the slow work.

The pattern: the webhook handler claims the row (atomic `update ... returning`), responds 200 immediately, and the slow work (download + Gemini call + embed + update) runs in `after()`. If the function times out mid-work, the row stays in `processing` and a periodic cron sweep flips stuck rows back to `pending`.

### 4.2 Supabase Database Webhook

Supabase dashboard → Database → Webhooks. Create a webhook on `clips`:

- Events: `INSERT`
- URL: `https://yourapp.com/api/embedder/tick`
- Headers: `x-webhook-secret: <EMBEDDER_WEBHOOK_SECRET>`

The webhook just pings the endpoint with the new row; the endpoint doesn't trust the payload, it re-reads the row from Postgres.

### 4.3 `/api/embedder/tick`

```ts
// app/api/embedder/tick/route.ts
import { after } from 'next/server';
import { createClient } from '@supabase/supabase-js';
import { GoogleGenerativeAI } from '@google/generative-ai';

export const runtime = 'nodejs';
export const maxDuration = 60;

export async function POST(req: Request) {
  if (req.headers.get('x-webhook-secret') !== process.env.EMBEDDER_WEBHOOK_SECRET) {
    return new Response('unauthorized', { status: 401 });
  }

  const body = await req.json();
  const clipId = body.record?.id;
  if (!clipId) return new Response('no id', { status: 400 });

  // after() runs after the response is sent — keeps the function alive
  // for the slow Gemini work without blocking the 200 response to Supabase.
  after(processClip(clipId).catch((err) => console.error('embed failed', clipId, err)));

  return Response.json({ ok: true, queued: true });
}
```

### 4.4 `processClip` — the actual work

```ts
async function processClip(clipId: string) {
  const supabase = createClient(/* service role */);

  // Atomically claim the row
  const { data: claimed } = await supabase
    .from('clips')
    .update({ embedding_status: 'processing' })
    .eq('id', clipId)
    .eq('embedding_status', 'pending')
    .select('id, storage_path')
    .single();
  if (!claimed) return;   // someone else got it, or already done

  try {
    // 1. Download clip
    const { data: blob } = await supabase.storage
      .from('gameplay-clips')
      .download(claimed.storage_path);
    const buffer = Buffer.from(await blob!.arrayBuffer());

    // 2. Ask Gemini for a caption
    const genai = new GoogleGenerativeAI(process.env.GEMINI_API_KEY!);
    const model = genai.getGenerativeModel({ model: 'gemini-2.0-flash' });

    const captionRes = await model.generateContent([
      {
        inlineData: {
          mimeType: 'video/webm',
          data: buffer.toString('base64'),
        },
      },
      {
        text: `Describe the physical action in this short gameplay clip in one or two sentences.
Focus on body movement: punches (jab/cross/hook), dodges, kicks, combos, pace, intensity.
Do not describe the background or UI. Output plain text, no preamble.`,
      },
    ]);
    const caption = captionRes.response.text().trim();

    // 3. Embed the caption
    const embedModel = genai.getGenerativeModel({ model: 'text-embedding-004' });
    const embedRes = await embedModel.embedContent(caption);
    const embedding = embedRes.embedding.values;   // number[768]

    // 4. Write back
    await supabase
      .from('clips')
      .update({
        caption,
        embedding,
        embedding_status: 'ready',
      })
      .eq('id', clipId);

  } catch (err) {
    await supabase
      .from('clips')
      .update({ embedding_status: 'failed' })
      .eq('id', clipId);
    throw err;
  }
}
```

### 4.5 Sweep for stuck rows

Add a Vercel Cron (or Supabase scheduled function) every 5 minutes:

```sql
update clips
set embedding_status = 'pending'
where embedding_status = 'processing'
  and created_at < now() - interval '10 minutes';
```

This recovers clips whose processing was killed by a function timeout.

### 4.6 Caption prompt quality

The caption is what gets embedded, so its quality is what makes semantic search work. A few things that matter:

- Keep prompts focused on physical action. Gemini will happily describe the player's shirt if you let it, and that pollutes the embedding space with irrelevant variance.
- Keep captions short and concrete. One or two sentences, no hedging.
- Use consistent vocabulary. If Gemini sometimes says "threw a punch" and sometimes "delivered a strike," semantically similar clips will end up further apart in embedding space. You can either (a) post-process captions through a normalization pass or (b) give Gemini a small controlled vocabulary in the prompt. Start with (b).

Iterate on the prompt with a handful of real clips before turning the webhook on for everything.

## Phase 5 — Query Planner

Note: `lib/search/index.ts` already exists as an empty stub. The dispatcher and planner logic built in Phases 5–6 lives there.

This is the component that translates natural-language questions into structured plans. Gemini **does not write SQL**. It picks from a fixed set of plan shapes.

### 5.1 Plan shapes

```ts
type AggregatePlan = {
  kind: 'aggregate';
  metric: 'count' | 'sum' | 'avg' | 'max';
  target: 'match_events' | 'match_summaries';
  filters: {
    eventType?: string;
    playerId?: string;
    matchId?: string;
    since?: string;          // ISO timestamp
    until?: string;
  };
  groupBy?: 'match' | 'event_type' | 'day';
  orderBy?: 'metric_desc' | 'metric_asc';
  limit?: number;
};

type RetrievePlan = {
  kind: 'retrieve';
  filters: {
    eventType?: string;
    minCount?: number;         // e.g. "at least 5 punches" in the clip
    playerId?: string;
    matchId?: string;
    since?: string;
    until?: string;
  };
  semanticQuery?: string;      // if present, ranked by similarity
  limit?: number;
};

type HybridPlan = {
  kind: 'hybrid';
  filters: RetrievePlan['filters'];
  semanticQuery: string;       // required for hybrid
  limit?: number;
};

type QueryPlan = AggregatePlan | RetrievePlan | HybridPlan;
```

### 5.2 Prompting Gemini

Use `gemini-2.0-flash` with JSON mode and a schema. The system prompt gives it:

1. A description of the three plan shapes.
2. The allowed values for `metric`, `target`, `eventType`, etc.
3. Three or four worked examples covering aggregate / retrieve / hybrid.
4. An instruction to pick the simplest plan that answers the question.

A sketch:

```ts
const SYSTEM = `
You convert a player's natural-language question into a structured query plan for a
gameplay search system. Output ONLY a JSON object matching one of three shapes:
aggregate, retrieve, or hybrid.

Use "aggregate" for questions about counts, totals, averages, or rankings.
Use "retrieve" for questions that ask for clips matching exact structured criteria.
Use "hybrid" when the question combines a structured filter with a fuzzy/descriptive
concept like "amazing", "fast", "powerful", "sloppy".

Allowed event types: punch, dodge, kick, block.
Current player id: {playerId}.

Examples:
Q: "How many punches have I thrown this session?"
A: {"kind":"aggregate","metric":"count","target":"match_events","filters":{"eventType":"punch","playerId":"{playerId}","since":"{sessionStart}"}}

Q: "Show me a clip where I did 10 punches."
A: {"kind":"retrieve","filters":{"eventType":"punch","minCount":10,"playerId":"{playerId}"},"limit":5}

Q: "Find a clip where I threw an amazing combo."
A: {"kind":"hybrid","filters":{"eventType":"punch","minCount":3,"playerId":"{playerId}"},"semanticQuery":"amazing combo attack","limit":5}

Q: "What was my best round?"
A: {"kind":"aggregate","metric":"max","target":"match_summaries","filters":{"playerId":"{playerId}"},"orderBy":"metric_desc","limit":1}
`;
```

Validate Gemini's output against a Zod schema before handing to the dispatcher. If parsing fails, return a friendly "I didn't understand that" rather than throwing.

export async function POST(req: Request) {

## Phase 6 — Query Dispatcher

Executes plans against a fixed set of parameterized SQL templates. No dynamic SQL construction from user input.

### 6.1 Aggregate dispatch

```ts
async function runAggregate(p: AggregatePlan) {
  if (p.target === 'match_events' && p.metric === 'count') {
    // head: true means data is null — the count comes from the count field.
    const { count, error } = await supabase
      .from('match_events')
      .select('*', { count: 'exact', head: true })
      .eq('player_id', p.filters.playerId!)
      .eq('event_type', p.filters.eventType!)
      .gte('occurred_at', p.filters.since ?? '1970-01-01');
    return { answer: `${count ?? 0}`, count: count ?? 0 };
  }

  if (p.target === 'match_summaries' && p.orderBy === 'metric_desc') {
    const { data } = await supabase
      .from('match_summaries')
      .select('*')
      .eq('player_id', p.filters.playerId!)
      .order('event_totals->punch', { ascending: false })
      .limit(p.limit ?? 1);
    return { answer: 'Your best round:', matches: data };
  }

  // ...other aggregate shapes
}
```

Keep each shape boring. If you find yourself writing a dynamic SQL builder, stop — add another named template instead.

### 6.2 Retrieve dispatch

```ts
async function runRetrieve(p: RetrievePlan) {
  let q = supabase
    .from('clips')
    .select('id, storage_path, caption, event_counts, started_at');

  if (p.filters.playerId)  q = q.eq('player_id', p.filters.playerId);
  if (p.filters.matchId)   q = q.eq('match_id', p.filters.matchId);
  if (p.filters.since)     q = q.gte('started_at', p.filters.since);

  if (p.filters.eventType && p.filters.minCount) {
    // event_counts->>'punch' >= N
    q = q.gte(`event_counts->>${p.filters.eventType}`, p.filters.minCount);
  }

  q = q.limit(p.limit ?? 5);
  const { data } = await q;
  return { clips: await withSignedUrls(data ?? []) };
}
```

### 6.3 Hybrid dispatch

This is where pgvector earns its keep. Filter structurally first, then rank by cosine distance. Use an RPC function so you can express the order-by cleanly:

```sql
create or replace function search_clips_hybrid(
  p_player_id uuid,
  p_event_type text,
  p_min_count  int,
  p_query_vec  vector(768),
  p_limit      int
) returns table (
  id uuid, storage_path text, caption text,
  event_counts jsonb, distance float
) language sql stable as $$
  select id, storage_path, caption, event_counts,
         embedding <=> p_query_vec as distance
  from clips
  where player_id = p_player_id
    and embedding_status = 'ready'
    and (p_event_type is null
         or (event_counts->>p_event_type)::int >= coalesce(p_min_count, 1))
  order by embedding <=> p_query_vec
  limit p_limit;
$$;
```

Dispatcher:

```ts
async function runHybrid(p: HybridPlan) {
  // Embed the user's semantic query with the SAME model used for captions
  const embed = await geminiEmbed(p.semanticQuery);

  const { data } = await supabase.rpc('search_clips_hybrid', {
    p_player_id: p.filters.playerId,
    p_event_type: p.filters.eventType ?? null,
    p_min_count: p.filters.minCount ?? null,
    p_query_vec: embed,
    p_limit: p.limit ?? 5,
  });

  return { clips: await withSignedUrls(data ?? []) };
}
```

Critical invariant: the model embedding the query at search time must be the same one that embedded the captions at ingest time. Mixing models silently produces garbage results — the vectors live in different spaces.

### 6.4 Signed URLs

Never return raw storage paths to the client. Generate short-lived signed URLs:

```ts
async function withSignedUrls(clips: any[]) {
  return Promise.all(clips.map(async (c) => {
    const { data } = await supabase.storage
      .from('gameplay-clips')
      .createSignedUrl(c.storage_path, 300);  // 5 min
    return { ...c, videoUrl: data?.signedUrl };
  }));
}
```

### 6.5 Natural-language wrapper

After the dispatcher returns, optionally feed the result back to Gemini with the original question to produce a one-line natural-language answer ("You threw 47 punches this session, 12 more than last time"). This is cosmetic — don't gate correctness on it.

## Phase 7 — Hardening

The things that separate a demo from a system.

### 7.1 Metrics to watch

- **Ingestion lag**: time from `ChunkReady` to `clips` row insert. Should be sub-second.
- **Embedding lag**: time from clip insert to `embedding_status = 'ready'`. Target under a minute; fine up to a few.
- **Failed clips**: count of `embedding_status = 'failed'` per day. Should trend to zero.
- **Query latency**: p50/p95 on `/api/query`. Aggregates should be single-digit ms; hybrids depend on vector index health.

Log these to whatever you're already using (Vercel Analytics, Axiom, a Grafana you already run).

### 7.2 Retry and backoff

- Client-side: retry failed chunk uploads and event batches with exponential backoff. Cap at 5 tries then drop on the floor (and log — losing a chunk is annoying but not catastrophic, losing an event batch slightly undercounts).
- Embedder: `failed` rows get retried by the sweep cron up to N times; after N, mark them `abandoned` so you know to investigate.

### 7.3 Cost control

Gemini video captioning isn't free. Guardrails:

- Only caption clips where `event_counts` has at least one non-zero entry. A chunk with zero detected actions is almost certainly not worth embedding.
- Rate-limit the embedder to something reasonable per user per hour.
- Make match_summaries the primary target for aggregate queries — they're tiny and don't require scanning raw events.

### 7.4 Privacy

Clips are video of the player. Treat them that way:

- Private bucket, signed URLs only.
- Delete-user endpoint that wipes clips, events, summaries, and storage objects.
- Don't log caption text outside the DB — it's a description of the user's body.

### 7.5 Index tuning

Once you have real data:

```sql
-- Check how many lists ivfflat should use
-- Rule of thumb: sqrt(rows) for <1M rows
analyze clips;
-- If queries are slow, rebuild with a better lists value:
drop index clips_embedding_idx;
create index clips_embedding_idx on clips
  using ivfflat (embedding vector_cosine_ops) with (lists = <sqrt(rowcount)>);
```

For ivfflat, also set `ivfflat.probes` per session for recall/latency tradeoff (default 1, try 10 for higher recall).

## Build order recap

1. **Phase 0**: schema, storage, env. Half a day.
2. **Phase 1**: EventTracker. You have punch tracking already; extend the pattern. One to two days depending on how many gesture types you want at launch.
3. **Phase 2**: `/api/clips/upload` and `/api/match-events`. Half a day.
4. **Phase 3**: wire recorder + event flush. Half a day. **At this point, count-based queries work end-to-end.** Ship a dashboard that just shows "X punches this session" to validate.
5. **Phase 4**: embedder webhook. One to two days with prompt iteration.
6. **Phase 5**: query planner. One day.
7. **Phase 6**: dispatcher + RPC. One to two days.
8. **Phase 7**: hardening, ongoing.

The critical rule: don't work on Phases 5–6 until Phase 4 is producing good captions. Bad captions make the hybrid path feel broken even when the code is correct, and you'll waste time debugging SQL when the actual bug is the prompt.

## What you can skip or defer

- **`match_summaries` live updates**: write-on-end is fine. Skip incremental.
- **Event idempotency IDs**: add later if duplicates become visible.
- **Generated columns for punch counts**: add only if GIN containment queries become slow.
- **A proper queue for the embedder**: Next.js + waitUntil + sweep is enough until you have real throughput.
- **Cross-user / public clips**: the whole plan assumes per-player. Broadening later is additive — add columns for visibility, adjust RLS, done.
